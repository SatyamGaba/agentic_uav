# Agentic UAV Swarm Framework and Simulation Plan
Created by Satyam Gaba

## Purpose of this document

This document summarizes the system for the paper: a decentralized UAV swarm architecture in which each UAV hosts an onboard AI agent for high-level mission reasoning and swarm coordination, while conventional planning and flight-control modules handle geometry, motion, and stabilization.

The goal is to give the team a clear, shared understanding of:

1. what problem we are solving,
2. what the proposed system looks like,
3. what the agent does and does not do,
4. how communication works within and across UAVs,
5. and how we can evaluate the framework quickly in simulation for a paper.

---

# 1. Problem statement

Current UAV swarms are often built around one of two paradigms:

- centralized coordination, where a single controller or planner assigns roles and tasks, or
- fixed decentralized logic, where each UAV follows pre-programmed rules and coordination protocols.

These approaches work well in structured settings, but they become brittle when the environment changes unexpectedly. In realistic missions, the swarm may face:

- UAV failure or dropout,
- degraded perception due to smoke, dust, or occlusion,
- communication loss or intermittent connectivity,
- changing mission priorities,
- newly detected hazards or blocked regions,
- mismatch between the original plan and the current state of the environment.

Our core idea is to give each UAV an onboard agent that can reason over mission goals, local observations, peer updates, and internal health status to make high-level decisions such as role changes, task reallocation, or replanning requests. The agent is not responsible for direct control or low-level collision avoidance. Instead, it sits above the traditional planning and control stack.

This architecture is intended to improve resilience, flexibility, and mission continuity in dynamic environments while keeping the safety-critical control loop conventional.

# 2. Scope of the paper

The paper should position the contribution clearly.

## What our paper is about

The paper proposes a framework for decentralized, agent-based UAV swarms where:

- each UAV is a self-contained mission-level agent,
- swarm coordination is distributed rather than centrally controlled,
- communication is structured and lightweight,
- the agent reasons at the mission and tactical level,
- existing planning and autopilot modules handle motion generation and control,
- the framework is evaluated in simulation.

## What our paper is not trying to do

The paper is not claiming that an LLM directly controls drone motors or replaces flight control.

The paper is also not trying to invent a completely new consensus, auction, or swarm-control algorithm from first principles. Instead, we assume that established coordination methods can be used as tools within the broader agentic architecture.

This distinction is important and should be stated explicitly in the paper.

# 3. High-level system view

The system has two levels:

## Level 1: Intra-UAV intelligence

Each UAV has an internal architecture consisting of:

- sensors, perception and state estimation, local world model (generated by perception)
- communication manager,
- health and resource monitor,,
- agent core with tool interfaces,
- safety governor,
- flight planner/autopilot and execution stack. (w/ safety governor)

This is the internal decision-making loop for a single UAV.

## Level 2: Inter-UAV coordination

Across the swarm, UAVs exchange structured messages such as:

- heartbeat and health summaries,
- task bids and task commitments,
- intent summaries, map or observation updates,
- hazard alerts,
- degradation or failure notices.

This is the coordination layer that allows the swarm to function as a team.

The architecture should therefore be described as a combination of local autonomy and distributed cooperation.

# 4. Architecture of a single UAV

This section describes the blocks inside one UAV. Important ones are Perception, communication, agent core, flight planner, control - we may merge or omit others but keeping it here for understanding

## 4.1 Mission specification interface

This block represents the mission-level information that is available to the swarm.

### Inputs

- user-provided mission goal,
- mission constraints,
- geographic restrictions,
- mission priorities,
- completion criteria.

### Outputs

- machine-readable mission specification,
- task objectives,
- priority weights,
- mission phase information.

This can be distributed to all UAVs before mission start and updated if needed.

---

## 4.2 Perception and state estimation

This block converts raw sensor data into actionable state information.

### Inputs

- camera streams,
- IMU measurements,
- GNSS or localization signals,
- radar, lidar, or range sensor data,
- actuator feedback.

### Outputs (same as traditional systems)

- own pose and velocity,
- local obstacle detections,
- target or object detections,
- local free-space or map information,
- perception confidence,
- hazard indicators.



The output will covey a significant event to the Agent in case of a significant event. *( Format of message TBD)*

### Role in the system

This block continuously supports the lower-level flight and planning stack. It may also generate significant event flags when it detects something strategically significant, such as a blocked region, degraded visibility, or a potential target.

The important point is that the agent does not receive raw sensor streams. It receives structured summaries from perception when a significant event is triggered.

---

## 4.3 Communication manager

This block handles incoming and outgoing inter-UAV messages.

### Inputs

- peer messages from the wireless network,
- outgoing message requests from the agent,
- message priorities,
- link quality information.

### Outputs

- validated and decoded peer updates,
- neighbor state summaries,
- forwarded or broadcast messages,
- communication-related event triggers.

### Role in the system

The communication manager is responsible for packaging and routing information between UAV agents. It also decides which messages are urgent enough to propagate broadly and which should remain local.

This block can also trigger the agent when important peer-originated events occur, such as:

- a nearby UAV failure,
- a hazard alert,
- a task reassignment request,
- loss of relay coverage,
- change in neighbor availability.

---

## 4.4 Health and resource monitor

This block tracks the UAV's internal capability state.

### Inputs

- battery or energy level,
- CPU and memory usage,
- sensor health,
- radio quality,
- autopilot status,
- fault codes.

### Outputs

- resource summary,
- degradation status,
- availability level,
- remaining operating budget,
- internal warning flags.

### Role in the system

This is important because the agent must reason not just about the external environment, but also about whether the UAV is still capable of taking on or continuing a role.

---

## 4.5 Local world model

This is the UAV's structured internal representation of the current situation.

### Inputs

- mission specification,
- perception summaries,
- peer updates from communication,
- own health and resource state,
- execution feedback from planning and autopilot.

### Outputs

- local belief state,
- team state estimate,
- known hazard map,
- task status view,
- uncertainty estimates,
- context summary for the agent.

### Role in the system

The local world model is the immediate context the agent reasons over. It should be compact, structured, and continuously updated. It serves as the memory layer between sensing, communication, and decision-making.

---

## 4.6 Agent core

This is the mission-level reasoning component of the UAV.

### Inputs

- summarized local world model,
- mission goals and constraints,
- current role and task state,
- peer status and peer intents,
- internal health and resource status,
- event triggers from perception or communication.

### Outputs

- proposed high-level decision,
- role-change decision,
- task bid or acceptance,
- message intent,
- tool invocation request,
- replan or contingency trigger,
- confidence estimate.

### What the agent does

The agent is responsible for high-level reasoning, such as:

- deciding whether to keep or change role,
- requesting participation in a task,
- initiating replanning after a disruption,
- summarizing local information for peer sharing,
- deciding whether to continue, regroup, or hand off a responsibility,
- deciding when additional coordination is needed.

### What the agent does not do

The agent does not:

- generate motor commands,
- perform continuous control,
- implement low-level collision avoidance,
- replace path planning,
- replace the autopilot.

This distinction should be made very explicit in the paper.

---

## 4.7 Tool interfaces

The agent does not solve every subproblem directly. It should invoke specialized tool modules.

### Example tools

- task allocation or bid-generation module,
- route or waypoint planner,
- mission progress checker,
- hazard assessment or prioritization module,
- peer-query or communication module,
- contingency planner.

### Inputs

- structured request from the agent,
- world model state,
- mission constraints,
- relevant cost functions.

### Outputs

- candidate task allocation result,
- route or waypoint plan,
- completion estimate,
- priority score,
- communication plan,
- feasibility result.

### Role in the system

The tools perform deterministic or algorithmic functions that are better handled by classical methods. The agent chooses when and why to use them.

---

## 4.8 Safety governor

This block ensures that high-level decisions are operationally safe before execution.

### Inputs

- proposed task or trajectory,
- current safety state,
- geofence rules,
- flight envelope constraints,
- emergency policies.

### Outputs

- approved command,
- modified safe alternative,
- rejection with fallback behavior.

### Role in the system

The safety governor acts as a protective boundary between mission-level decisions and vehicle execution.

---

## 4.9 Autopilot and execution layer

This is the conventional flight-control and motion-execution layer.

### Inputs

- approved waypoint list,
- trajectory or local target,
- mission mode commands.

### Outputs

- actuator-level control,
- execution feedback,
- tracking status,
- mission progress feedback.

### Role in the system

This layer performs the low-level control. It is assumed to already exist and to remain responsible for stabilization and continuous motion execution.

# 5. Internal reasoning loop of the agent

The reasoning loop should be described as an event-aware **sense-think-act** cycle.

## 5.1 Sense

The agent receives structured inputs from:

- the local world model,
- communication manager,
- health monitor,
- mission progress state.

These are not raw sensor streams but meaningful summaries.

## 5.2 Triggering conditions

The agent does not need to run at the same rate as the control loop. Instead, it can be triggered by two mechanisms:

### Periodic trigger

A low-rate periodic reassessment cycle. This ensures the UAV revisits its role and mission state even if no major event has been flagged.

### Event-driven trigger

Immediate invocation when a significant change is observed. Examples include:

- new hazard detected,
- peer failure reported,
- loss or recovery of connectivity,
- task completion,
- inability to continue current role,
- mission priority change,
- degraded visibility or sensing.

This hybrid triggering mechanism is important because it balances responsiveness with computational efficiency.

## 5.3 Think

In the think stage, the agent:

- checks current mission relevance,
- compares current role against updated needs,
- assesses whether new coordination is required,
- decides if a tool should be called,
- decides what information should be shared,
- evaluates whether it should keep course, request reassignment, or trigger replanning.

The think stage should be described as deliberative reasoning rather than raw control.

## 5.4 Act

The output of the agent is a structured decision package that can include:

- a task bid,
- a role change proposal,
- a request to call a planner,
- a communication request,
- a new intent summary,
- a contingency request.

The output is then passed to the appropriate subsystem rather than directly to the motors.

# 6. Inter-UAV coordination model

The swarm should be described as a decentralized, peer-to-peer system with optional multi-hop message propagation.

## 6.1 Communication philosophy

Not every message needs to be shared with the entire swarm. Communication should be selective and structured.

A practical policy is:

- routine heartbeat and status updates go to nearby peers,
- local task negotiation occurs among relevant neighbors,
- urgent hazard or failure alerts can propagate wider,
- important summary information can be relayed through the network.

This is sufficient for the paper. We do not need to over-specify the exact network protocol unless we choose to include a dedicated communication subsection.

## 6.2 Types of shared messages

### Heartbeat and status

Includes:

- UAV identity,
- current role,
- current position or region,
- health status,
- energy summary,
- communication quality.

### Task bid or commitment

Includes:

- task identifier,
- willingness or suitability score,
- expected feasibility,
- role proposal,
- commitment state.

### Intent summary

Includes:

- current local objective,
- near-term plan,
- role status,
- expected contribution.

### Hazard alert

Includes:

- hazard type,
- location or affected region,
- urgency,
- timestamp,
- confidence.

### Failure or degradation notice

Includes:

- UAV unavailable or degraded,
- role abandonment,
- sensor degradation,
- low-energy retreat state,
- communication loss recovery.

### Observation or coverage update

Includes:

- region already covered,
- new object or target detection,
- blocked area,
- updated environmental state.

## 6.3 Shared allocation state

A natural question is how multiple UAVs avoid taking the same role or task.

The paper should explain that each UAV maintains a partial shared view of:

- what tasks exist,
- which tasks are currently assigned,
- which roles are already sufficiently covered,
- which UAVs are available.

The exact assignment may be determined using an existing distributed method such as a consensus or auction-based protocol. Our contribution is not the coordination algorithm itself, but how the onboard agent reasons about when to use it and how to respond to its results.

# 7. Why use an agent at all?

This section is important because a reviewer may ask why a deterministic mission layer is not enough.

## 7.1 Baseline criticism

A reasonable criticism is that all high-level decisions could be handled by deterministic logic and rule-based state machines.

## 7.2 Our answer

The argument for an agentic layer is not that it replaces all classical logic. The argument is that it improves flexibility in situations where:

- the mission specification is open-ended,
- the environment changes in ways not fully enumerated beforehand,
- priorities change during execution,
- multiple competing factors must be balanced,
- local and peer information must be integrated into a higher-level decision.

The agent is especially useful as a mission-level reasoning layer that can:

- interpret broader context,
- choose among available coordination actions,
- summarize and communicate relevant information,
- adapt role reasoning under uncertainty,
- provide a general interface between mission intent and task-level tools.

The paper should present the agent as a flexible decision orchestrator rather than a replacement for classical autonomy.

# 8. Paper-facing description of the proposed framework

This section can almost be reused directly in the paper.

## Proposed framework summary

We propose a decentralized UAV swarm framework in which each UAV hosts an onboard mission-level agent that reasons over local observations, peer updates, mission goals, and internal resource state to make high-level coordination decisions. The agent does not perform direct control. Instead, it invokes established planning and coordination tools, communicates structured summaries to peers, and submits high-level actions through a safety-governed execution stack. This architecture separates strategic reasoning from motion generation and control, enabling greater resilience to UAV loss, perception degradation, and communication disruptions while maintaining compatibility with conventional autonomy modules.

# 9. Experiment section

This section outlines a simulation-first evaluation plan that is practical for us to implement.

## 9.1 Goal of experiments

The experiments should demonstrate the value of the proposed architecture without requiring real UAV deployment.

We want to show that compared with a classical baseline, the proposed agentic swarm:

- adapts better to disruptions,
- maintains mission progress under failures,
- reallocates roles more effectively,
- remains functional under communication degradation,
- provides a better balance between decentralization and mission continuity.

## 9.2 Evaluation philosophy

We should keep the experiments simple enough to implement quickly, but structured enough to support a strong paper.

A lightweight 2D or abstract multi-agent simulator is enough for the main evaluation. We do not need to model full aerodynamics unless we later decide to add a small demonstration in a robotics simulator.

The simulation should focus on decision-making and coordination.

---

## 9.3 Core simulated scenario

A suitable scenario for the paper is a disaster-response or search-and-assist mission over a bounded area.

### Example mission

A swarm of UAVs is deployed to inspect and map a disaster zone, maintain communication coverage, and identify designated areas of interest.

### Initial deployment assumption

For the main simulation experiments, we explicitly assume that all UAVs are initially deployed from the same or nearby sector. This reflects missions launched from a common base, depot, command vehicle, charging station, or staging area. The assumption should be stated as part of the experimental setup, because other UAV swarm studies may use distributed, random, predefined, or optimized initial positions.

### Dynamic events introduced during the mission

- one UAV drops out,
- one sector becomes hazardous or inaccessible,
- local visibility degrades in part of the environment,
- communication quality decreases in a region,
- a new task or priority region appears mid-mission.

This scenario is good because it is intuitive, non-controversial, and highlights the strengths of decentralized adaptation.

---

## 9.4 Experimental conditions

We should vary at least the following:

### Swarm size

For example:

- 5 UAVs,
- 10 UAVs,
- 15 UAVs.

### Communication condition

For example:

- ideal communication,
- intermittent packet loss,
- localized communication blackout.

### Failure condition

For example:

- no UAV loss,
- single UAV dropout,
- two UAV degradations.

### Environment condition

For example:

- normal perception,
- partial visibility degradation,
- newly blocked region.

This gives enough diversity without making implementation too large.

---

## 9.5 Baselines

The final comparison set should include three baselines and the proposed decentralized agentic method.

### Baseline A: Centralized static assignment

A classical mission controller assigns tasks and roles at mission start. Reallocation is limited or delayed.

### Baseline B: Decentralized rule-based system

Each UAV follows fixed rules for role switching and communication without an agentic reasoning layer.

### Baseline C: Decentralized task-consideration scheduling

Each UAV uses a modern non-agentic decentralized scheduler inspired by Chen, Li, and Peng (2023), "Decentralized UAV Swarm Scheduling with Constrained Task Exploration Balance." This baseline should score candidate tasks locally, balance low-cost or nearby task selection with completion of neighboring tasks, and resolve conflicts through explicit communication or consensus-style updates.

### Proposed method

Each UAV uses the proposed decentralized mission-level agent along with the same planning and execution backbone. The agent reasons over local observations, peer messages, mission goals, and internal state to choose roles, targets, and communication actions.

This baseline structure is important because it isolates the benefit of the agentic reasoning layer against centralized static planning, fixed decentralized rules, and a stronger modern decentralized scheduling method.

CBBA and PI should be discussed as classical task-allocation ancestors in related work, but the main experiments should prioritize the newer task-consideration scheduler as Baseline C when implementation time allows.

## 9.6 Metrics

### Adaptability

Measure how quickly the swarm responds after a disruption.

Possible operationalization:

- time from event occurrence to first reassignment,
- time from event occurrence to stable new task allocation,
- number of cycles needed to recover mission progress.

### Task completion time

Measure the total time required to complete the mission or mission subgoals.

Possible operationalization:

- mission finish time,
- average time per assigned task,
- time to complete a newly introduced task.

### Mission success rate

Measure whether the mission objective was achieved despite disruptions.

Possible operationalization:

- percentage of trials where required coverage was achieved,
- percentage of trials where critical tasks were completed,
- mission-completion ratio under failures.

### Role reallocation effectiveness

Measure whether the swarm can redistribute work efficiently.

Possible operationalization:

- number of abandoned tasks recovered,
- percentage of tasks successfully reassigned,
- imbalance in task distribution after reallocation.

### Communication overhead

Measure whether the architecture is communication-efficient.

Possible operationalization:

- number of inter-UAV messages,
- average message size,
- event-triggered communication count,
- total communication load per mission.

### Resilience under communication loss

Measure how much performance degrades when links are unreliable.

Possible operationalization:

- mission success versus packet loss,
- adaptation delay under blackout conditions,
- proportion of local decisions made without global synchronization.

### Decision latency

Measure the time from event detection to decision output by the agent.

Possible operationalization:

- average event-to-decision delay,
- distribution of decision times by event type,
- comparison between agentic and rule-based response latency.

## 9.7 Suggested experimental studies

### Experiment 1: UAV dropout recovery

Inject the loss of one UAV during the mission.

Measure:

- time to detect role gap,
- time to reassign abandoned task,
- final mission completion rate.

Expected result:
The agentic swarm should recover faster and complete a larger portion of the mission.

### Experiment 2: Communication degradation

Introduce packet loss or region-specific communication blackout.

Measure:

- communication overhead,
- time to recover coordination,
- mission success under degraded links.

Expected result:
The proposed system should maintain acceptable performance because each UAV retains local autonomy.

### Experiment 3: Dynamic task insertion

Introduce a new urgent region or task mid-mission.

Measure:

- time to incorporate new task,
- task reassignment quality,
- degradation in original mission progress.

Expected result:
The agentic system should handle reprioritization more gracefully than a static baseline.

### Experiment 4: Perception degradation

Mark one region as degraded visibility or uncertain sensing.

Measure:

- role switching behavior,
- reassignment toward healthier UAVs,
- mission completion ratio.

Expected result:
The agentic swarm should redistribute responsibilities according to capability.

## 9.8 Minimal implementation path

To keep effort low, the implementation can be staged.

### Stage 1: Simple custom simulator

Build a 2D multi-agent simulator with:

- agent positions,
- task regions,
- hazard regions,
- communication graph,
- event injection,
- message passing,
- role assignment state.

This should be the main evaluation platform.

### Stage 2: Plug in proposed reasoning policies

Implement:

- centralized baseline,
- rule-based decentralized baseline,
- agentic policy.

The agentic policy can use an LLM API or a simplified decision abstraction if necessary.

### Stage 3: Run repeated trials

For each condition, run multiple seeds and report averages and variance.

This is enough for a strong simulation-based evaluation section.

---

## 9.9 What not to overdo

To keep the work manageable, we should avoid the following for the first paper version:

- full 3D aerodynamics,
- real UAV hardware,
- complex low-level flight modeling,
- too many baselines,
- too many mission types,
- heavy communication-stack realism unless it directly supports the paper.

The core value is the decision and coordination framework, not full vehicle modeling.

# 11. Final takeaway for the team

The main claim of our paper should be simple and consistent:

We are not proposing that an LLM flies the drone. We are proposing that each UAV hosts an onboard mission-level agent that reasons about goals, context, failures, peer information, and task allocation, while relying on conventional planning and control modules for safe execution.

The experiments should therefore evaluate whether this additional mission-level intelligence improves swarm resilience and adaptability in dynamic simulated missions.

That is the framing that keeps the paper technically grounded, practically implementable, and sufficiently novel.
